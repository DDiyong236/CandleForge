# CandleForge Phase 2 — 데이터 길들이기 설계 문서

**작성일:** 2026-07-24
**분류:** Phase 2 설계 확정본 (Phase 1 Walking Skeleton 완료 후속)
**상태:** 착수 전 설계 확정. 구현 진행에 따라 갱신.

---

## 1. 목표

Phase 1은 KRW-BTC 1종목이 `수집→저장→조회`까지 관통하는 얇은 슬라이스였다. 그 결과 원본 체결이 **무한정 쌓이기만** 하는 상태다.

Phase 2는 이 프로젝트의 **정체성("대용량 시계열 데이터를 감당 가능한 크기로 길들이기")** 을 실현한다:

> **200종목으로 확대 → 배치로 빠르게 저장 → 캔들로 자동 집계 → 원본 24시간 뒤 삭제 + 캔들 압축** 을 통해 **"방치 시 월 수십 GB 무한증가 → 수 GB로 통제"** 를 실측으로 증명한다.

## 2. 증명하려는 역량

- 데이터 수명주기 설계 (다운샘플링 · 보관정책 · 압축)를 통한 저장 비용 통제
- 대용량 유입에 대한 저장 성능 최적화 (배치 insert)
- 기술 선택의 근거를 **측정으로 증명** (DB 크기 · 처리량 before/after)
- 외부 REST API 연동 (종목 목록 동적 조회)

## 3. 핵심 설계 결정 (Design Decisions)

### DD-1. 종목: 전체 KRW 마켓 (~200종목)
- Upbit REST `GET /v1/market/all`로 KRW 마켓 코드를 **동적 조회** (하드코딩 안 함).
- 이래야 초당 수백 건 유입 → "대용량 문제"가 실측으로 성립. 1종목(Phase 1)은 문제가 극적이지 않았음.

### DD-2. 캔들 집계: TimescaleDB 연속집계 (앱에서 직접 X)
- `time_bucket` + `first/last/max/min/sum`으로 OHLCV를 **SQL로 정의**, 갱신 정책으로 자동·증분 계산.
- 앱에 집계 로직을 두지 않음 → 재시작·지연도착·중복 정확성 문제를 DB에 위임. "왜 TimescaleDB인가"에 답이 됨.

### DD-3. 캔들 종류: 1분봉만 (`candles_1m`)
- 5분봉·시간봉은 소비처(지표 API)가 생기는 Phase 3에서 필요 시 추가 (YAGNI).
- 지금 만들면 소비처 없는 죽은 코드.

### DD-4. 보관/갱신 정책 (캔들 무결성 보장이 핵심)
| 설정 | 값 | 이유 |
|---|---|---|
| 원본 보관 (`drop_after`) | **24시간** | rolling ~2.6GB 고정. 최근 하루 원본 조회 가능 |
| 갱신 범위 (`start_offset`) | **1시간** | **삭제 영역을 절대 안 건드림** (1h < 24h). 지연도착 보정엔 충분 |
| 갱신 정지선 (`end_offset`) | **1분** | 진행 중인 미완성 봉 제외 |
| 갱신 주기 | **1분** | 최신성 확보 |

**불변 규칙: `start_offset(1h) < drop_after(24h)`.**
- 위반 시(갱신범위 > 보관기간): 갱신 작업이 "이미 삭제된 원본 구간"을 재계산하며 **멀쩡한 캔들을 빈 값으로 덮어써 손실**. (문서 확인된 함정)
- 준수 시: 갱신이 삭제 영역을 넘겨다보지 않아 안전. 또한 집계는 1분 내 완료되므로 24시간 뒤 삭제 시 이미 집계 완료 상태 → 이중 안전.

### DD-5. 압축: 캔들만, 7일 후
- 원본(`trades`)은 24h 뒤 어차피 삭제 + 실시간 삽입 중이라 압축 부적합 → 압축 안 함.
- 캔들(`candles_1m`)은 장기 보관 + 과거 봉은 확정(불변) → 압축 적합. **7일 지난 청크 압축.**
- 압축은 무손실. 시계열 특성(반복·delta)으로 큰 절감 기대하나 **실제 압축률은 측정으로 확인**.

### DD-6. 저장 최적화: 배치 insert
- Phase 1은 체결 1건당 INSERT 1회. 200종목(초당 수백 건)이면 DB 왕복이 병목 가능.
- 여러 건을 모아 한 번에 INSERT로 전환. **전환 전후 처리량을 측정**해 개선을 증명.
- 트레이드오프(모으는 지연, 크래시 시 버퍼 유실 위험)를 인지하고 설계.

### DD-7. 측정: 테이블 크기 SQL 조회 (before/after)
- `pg_total_relation_size` 등으로 `trades`·`candles_1m` 크기를 필요 시 조회.
- before(정책 미적용 추세) vs after(정책 적용 안정화) 크기를 README에 기록.
- 압축 전후 크기도 함께 기록 → "실측 N배 절감".

## 4. 아키텍처 변화 (Phase 1 → 2)

```
Phase 1:  WS(1종목) → 파싱 → [단건 저장] → trades (무한 증가)

Phase 2:
  REST /v1/market/all ──▶ 종목 목록(~200)
        │
  WS(200종목) → 파싱 → [배치 저장] → trades ──자동 집계──▶ candles_1m
                                       │(24h 삭제)         (장기보관, 7일후 압축)
                                       │
                                 DB 크기·처리량 측정
```

**책임 분담:**
- 앱: 종목 목록 조회 · 다중 구독 · 배치 저장 (수집·유입 담당)
- DB: 연속집계 · 보관정책 · 압축 (수명주기·저장 최적화 담당)

## 8. 위험과 대응

| 위험 | 대응 |
|---|---|
| 원본이 집계 전 삭제됨 | 집계 1분 완료 vs 삭제 24시간 → 압도적 여유. 갱신 작업 모니터링(Phase 5) |
| 갱신이 삭제구간 덮어 캔들 손실 | `start_offset(1h) < drop_after(24h)` 불변 규칙 준수 |
| 200종목 단건 insert 병목 | 배치 insert 전환, 측정으로 확인 |
| 배치 버퍼 크래시 시 유실 | 배치 크기·주기를 작게(예: 수백 건/수백 ms) 유지해 유실 창을 최소화 |
