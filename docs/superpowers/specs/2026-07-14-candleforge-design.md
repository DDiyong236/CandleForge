# CandleForge — 설계 문서 (Design Spec)

**작성일:** 2026-07-14
**분류:** 개인 포트폴리오 프로젝트 (백엔드 역량 어필)
**상태:** 착수 단계 설계 확정본. 개발 진행에 따라 갱신.

---

## 1. 한 줄 정의

실시간으로 쏟아지는 체결 데이터를 감당 가능한 크기로 "길들여" 저장하고, 그 위에 지표 조회와 조건 알림 서비스를 올린 백엔드 파이프라인.

> 이 프로젝트는 "주식 서비스를 만든다"가 아니라 **"대용량 스트리밍 파이프라인 문제를 푼다"**로 정의한다. 시세 데이터는 그 문제를 담는 그릇이다.

## 2. 목표 (증명하려는 백엔드 역량)

- 실시간 스트리밍 데이터의 안정적 수집과 재연결·멱등성 처리
- 데이터 수명주기 설계(다운샘플링·보관정책·압축)를 통한 저장 비용 통제
- 조회 성능 최적화(인덱싱·사전집계·캐싱)
- 이벤트 기반 비동기 처리(조건 매칭·알림 발송·재시도)
- 기술 선택의 근거를 설명하고 **측정으로 증명**하는 엔지니어링 사고

## 3. 제약 및 원칙

- **마감 없음.** 취업 지원 전까지 "보여줄 상태"를 만드는 것이 목표. 완성도·서사 품질 우선.
- **얇은 슬라이스 우선.** 계층별로 완성하지 않고, 1종목이 끝까지 관통하는 Walking Skeleton을 먼저 만든 뒤 각 단계를 두껍게 채운다.
- **측정을 처음부터 심는다.** 각 단계가 before/after 숫자를 하나씩 만들어낸다. 측정 하네스를 Phase 5로 미루지 않는다.
- **정직한 기술 선택.** 각 도구에 대해 "왜 골랐고, 없으면 어떻게 되는가"를 답할 수 있어야 한다. 과잉설계를 피한다.

## 4. 아키텍처

```
  Upbit WebSocket (체결, 다수 종목)
        │
        ▼
  [Ingestor]  재연결 · 중복방지(멱등성)
        │
        ├─────────────(Phase 4부터)──────────▶ Kafka topic: trades
        │                                         │
        ▼ (Phase 1~3: 직결)                       ├──▶ [저장 소비자] ──▶ TimescaleDB
  TimescaleDB                                     └──▶ [알림 소비자] ──▶ 조건매칭 → 비동기 알림
   · 원본 hypertable (짧게 보관 후 삭제)
   · 연속집계 → 1분/5분 캔들 (장기 보관)
   · 보관정책(retention) + 압축(compression)
        │
        ▼
  [지표/차트 API] ──읽기──▶ Redis 캐시 (인기 구간 · 사전집계)
   · 캔들 조회, 이동평균/변동성/거래량급증 계산
```

### 구성요소별 책임

| 구성요소 | 하는 일 | 의존 |
|---|---|---|
| Ingestor | Upbit WS 연결, 재연결, 중복방지, (P4부터) Kafka produce | Upbit WS, (P4) Kafka |
| 저장 소비자 | 체결 수신 → 원본 저장 + 캔들 집계 | TimescaleDB |
| 알림 소비자 | 체결 수신 → 등록 조건 매칭 → 비동기 알림 | (P4) Kafka, 알림 채널 |
| 지표/차트 API | 캔들 조회, 파생지표 계산, 캐싱 | TimescaleDB, Redis |

## 5. 기술 스택 및 선택 이유

| 영역 | 기술 | 선택 이유 |
|---|---|---|
| 프레임워크 | Spring Boot 3.x (Java 21) | 숙련도 높음 → 깊이 있는 구현에 집중 |
| 데이터 소스 | Upbit WebSocket (암호화폐) | 무료·무인증·24시간. 구조는 실주식(KIS API)과 동형이라 재활용 가능 |
| 저장소 | TimescaleDB (PostgreSQL 확장) | 하이퍼테이블·연속집계·보관정책·압축 내장. "데이터 길들이기" 서사와 정확히 맞물림 |
| 메시지 큐 | Kafka | **Phase 4에서 도입.** fan-out(저장+알림 소비자가 같은 스트림을 각자 소비) + 원본 보관정책과 스트림 소비의 분리. 아래 6절 참고 |
| 캐시 | Redis | 인기 조회 구간 캐싱, 사전집계 보조 |

## 6. 핵심 설계 결정 (Design Decisions)

### DD-1. Kafka는 Phase 4(알림 엔진)에서 도입한다

- **Phase 1~3은 `WS → DB 직결`.** 소비자가 저장 하나뿐일 때는 Kafka가 불필요하다(초당 수백 건은 배치 insert로 DB가 여유롭게 흡수).
- **두 번째 소비자(알림 매칭)가 실제로 생기는 Phase 4에서 Kafka를 도입.** 이때의 정당화 근거:
  1. **Fan-out** — 저장·알림 소비자가 같은 체결 스트림을 각자 독립적으로 전부 소비해야 한다. 컨슈머 그룹으로 깔끔하게 분리.
  2. **보관정책 분리** — 프로젝트의 정체성이 "원본을 빨리 삭제"인데, 알림 소비자가 DB 원본에 의존하면 "다 읽을 때까지 삭제 불가"가 되어 정책이 소비에 묶인다. 스트림(Kafka, 휘발)과 저장(DB, 영구)을 분리해 이 결합을 끊는다.
- "필요해져서 도입"이 git 히스토리로 증명된다. 앞 단계가 가벼워져 완주 확률도 높아진다.
- **기각한 대안:** (B) Kafka 미도입(폴링) — 더 단순하나 이력서에 스트림 처리 경험이 안 남음. (C) 처음부터 도입 — "왜 필요했나" 방어가 약함.

### DD-2. 종목 수는 Phase 2에서 수십~수백 개로 확대

- Phase 1은 1종목(예: KRW-BTC). Phase 2에서 Upbit KRW 마켓 다수(수십~수백)로 확대.
- 이래야 "월 수백 GB → 수 GB로 길들이기" 서사가 **실제 숫자**로 성립한다. 단일 종목은 초당 몇 건 수준이라 대용량 문제가 극적이지 않다.

### DD-3. 범위 자르기 — 알림 엔진(Phase 4)까지가 핵심

- 백테스팅·스크리너·실주식(KIS) 전환은 **"확장(취업 후/여유 시)"으로 명시적 분리.**
- 완성된 4단계 > 미완성 6단계.

### DD-4. 측정을 Phase 1부터 심는다

- 측정 하네스(부하 생성/리플레이 + 지표 수집)를 Phase 1에 심고, 각 단계가 before/after 숫자를 하나씩 만든다.
- README·면접의 핵심 이야깃거리가 여기서 나온다.

## 7. 핵심 기술적 도전

1. **저장 비용 통제** — 다운샘플링·보관정책·압축·대상종목 조정으로 DB를 수 GB~수십 GB로 유지.
2. **조회 성능** — 시간구간 집계 쿼리 병목을 인덱스·사전집계·캐싱으로 개선, 수치 비교.
3. **조건 매칭** — 등록된 다수 알림 조건을 실시간 스트림과 빠르게 대조.
4. **이벤트 비동기 처리** — Kafka fan-out, 비동기 발송, 중복방지·재시도.

## 8. 로드맵

| Phase | 목표 | 핵심 산출물 | 측정 숫자 |
|---|---|---|---|
| **0. 세팅** | 개발 기반 + GitHub 연결 | 폴더, git 연결·첫 push, Spring Boot 프로젝트, docker-compose(Timescale+Redis), README 뼈대, 기본 스키마 | — |
| **1. Walking Skeleton** | 1종목 끝까지 관통 | WS 수집 → DB 저장 → 조회 API 1개. 측정 하네스 심기 | 수집/저장 처리량 baseline |
| **2. 데이터 길들이기** ⭐ | 저장 비용 통제 | 다중 종목, 다운샘플링(1분/5분 캔들), 보관정책, 압축 | **DB 크기 before/after** |
| **3. 지표 API** | 조회 성능 | 캔들 조회 API, 파생지표(MA·변동성·거래량급증), Redis 캐싱·사전집계 | **조회 지연 before/after** |
| **4. 알림 엔진 + Kafka** | 이벤트 비동기 처리 | Kafka 도입(fan-out), 조건 등록, 실시간 매칭, 비동기 발송, 중복방지·재시도 | 매칭 지연, 발송 성공률 |
| **5. 마무리** | 서사 완성 | 부하 테스트, 모니터링·로깅, README에 문제·해결·측정 정리 | 종합 부하 수치 |
| *확장(후순위)* | — | *백테스팅 · 스크리너 · 실주식(KIS) 전환* | — |

**원칙:** 계층별로 완성하지 않고, 얇은 슬라이스를 먼저 관통시킨 뒤 각 단계를 두껍게 채운다.

## 9. 어필 포인트

"실시간 데이터를 수집해서, 감당 가능한 크기로 길들이고, 그 위에 조회·알림 서비스를 얹은 시스템" — 이 한 문장 안에 **실시간 수집 · 시계열 저장 최적화 · 조회 성능 튜닝 · 이벤트 기반 비동기 처리**라는 백엔드 채용 선호 키워드가 모두 담긴다.

## 10. 확장 가능성 (후순위)

- **실주식 전환:** 수집기만 증권사 API(KIS)로 교체하면 동일 구조.
- **백테스팅 엔진:** 축적된 과거 캔들 위에서 규칙 기반 매매를 결정론적으로 시뮬레이션.
- **종목 스크리너:** 다중 조건 필터링 종목 검색.
